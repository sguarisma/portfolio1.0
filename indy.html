The College Hill Independent


Making dropdown components more accessible, intuitive, and inclusive for all users.

Context

Over the course of two weeks, I focused on analyzing and redesigning a dropdown menu, prioritizing accessibility and usability. For this case study, I used the dropdown menu found in the navigation bar of a campus newspaper's website, The College Hill Independent.

Problem

Dropdown menus are one of the most common UI components used across web and desktop interfaces. Despite their simplicity, they often fall short in accessibility, efficiency, and keyboard compatability. This project explored how dropdowns interact with different user inputs and proposed a redesigned version that better support interactions.

Research

To begin, I analyzed dropdown implementations from three platforms:

  1. Brown University's website (browser-based dropdown)
  2. Visual Studio Code (desktop app dropdown)
  3. The College Hill Independent Website (my primary redesign focus)
I evaluate each by input type interactions (mouse, keyboard, touch), output cues (visual and screen reader support), and key UX factors, like learnability, memorability, and efficiency. Here were my key findings:

  • Brown University's website had the most accessible implementation of the dropdown menu as it fully supported all input types, full navigation with screen reader output. and had clear visual cues through borders, focus states, etc.
  • VSCode also supported multiple input types but lacked clear keyboard input usage, making it less intutive to use.
  • The College Hill Independent's website was the least accessible. Keyboard users couldn't fully navigate or select dropdown options, no clear hover outputs or focus indicators existed, and screen readers couldn't asccess dropdown items.

This analysis shaped my understanding of what features were essential for a truly accessible dropdown and where existing designs were falling short. Seeing the limitations in accessibility of the College Hill Independent's, or the Indy's, dropdown component, I decided to redesign it using what I had learned from my research.

Process

I created two state models of the Indy dropdown: one for mouse users and one for keyboard users. The original version only accounted for mouse input and lacked keyboard support almost entirely. Thus, I redeisgned both models to include accessible interaction pathways for all users. To better support non-mouse users and screen readers:

  1. I prioritized click-based triggers over hover-based ones (inspired by the VSCode dropdown component).
  2. I introduced both mouse and keyboard expand/collase functionality (inspired by the Brown University dropdown component).
  3. I added a clear focus order and consistent visual states.
  4. Used a two-column dropdown layout to reduce scrolling and increase readibility

Solution

My final dropdown redesign addressed both input and output shortcomings. These changes were implemented in Figma, where I designed both the initial and expanded states of the dropdown.

Changes made:

  • Added arrows beside the dropdown headings to indicate whether the menu was collapsed/expanded
  • Changed hover-based actions to click-based interactions
  • Added more visible active/hover states for subheadings
  • Limited the number of menu options to a two-column menu and added a "See All" option

Learnings

This project helped me recognize how often UI components are unintentionally designed in unaccessible manners , like with the prioritization of mouse-first assumptions without taking into account users who use screen readers or purely keyboard input. I took this observation into account for my final design by implementing multiple ways to expand the menu, including click-based interactions. My research and design process allowed me to realize the importance of clear feedback to users, keyboard accessibility, and how memorability and learnability are tied to consistency . In the future, I would love to develop my redesign and explore how other common components in the design process can be made more accessible.
